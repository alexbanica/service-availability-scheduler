# SPEC: Password Login And Reset Password

Status: Approved

## Purpose

Add password-based authentication and reset-password support to the existing JWT login flow while keeping reset delivery compatible with future email integration.

## Problem Statement

The application currently authenticates users by email only. That allows any caller who knows an existing email address to obtain a bearer token. The application needs password verification at login, a deterministic way for users to set or recover passwords, and a reset-token model that can later be connected to email delivery.

## Scope

- Require a password for `POST /api/login`.
- Persist password hashes for users.
- Manage checked-in database migrations for existing databases.
- Add a reset-password request flow protected by captcha.
- Persist reset-password tokens.
- Configure reset-token validity, defaulting to 1 hour.
- Enforce single active reset token per user.
- Mark reset tokens as used after a successful password reset.
- Log the reset URL server-side for now with a TODO noting that email delivery will replace logging later.
- Add backend APIs for captcha challenge, reset request, token validation, and password reset.
- Add frontend support for password login, forgot-password request, reset-token validation, setting a new password, and confirming the new password before submission.
- Preserve the existing JWT bearer token response contract after successful login.

## Out Of Scope

- Sending reset emails.
- Adding external captcha providers such as reCAPTCHA or hCaptcha.
- Adding OAuth, OIDC, SSO, passkeys, magic links, or MFA.
- Adding password change from an authenticated account settings page.
- Adding admin-managed password setup or password visibility into admin screens.
- Adding account lockout, password-history checks, breached-password checks, or complexity class rules.
- Migrating existing users through seed data or one-off scripts unless a later approved spec requires it.
- Changing workspace, reservation, service, owner, environment, role, or authorization semantics after authentication succeeds.
- Changing JWT signing, renewal, storage, or bearer-header behavior except where login input validation changes.
- Returning reset URLs through API responses.

## Definitions

- `Password login`: `POST /api/login` authenticates an existing user by email and password before issuing a JWT.
- `Password hash`: A one-way password verifier persisted for a user; raw passwords are never persisted or logged.
- `Reset-password token`: A random, unguessable, persisted token bound to one user and usable to set a new password before expiry.
- `Active reset token`: A reset token that is not used and has not expired.
- `Single active token`: At most one active reset token may exist for a user at any time.
- `Captcha challenge`: A first-party challenge generated by the application and solved before a reset link is requested.
- `Reset URL`: A URL path containing the reset token, using `/reset-password/:token`.

## Inputs And Constraints

- `POST /api/login` accepts `{ "email": string, "password": string }`.
- The password-reset submit API accepts `{ "token": string, "password": string, "confirm_password": string }`; camelCase `confirmPassword` may be accepted as a browser/client convenience if implemented consistently.
- Login continues to trim and lowercase email before lookup.
- The submitted password is treated as a secret and must not be logged.
- The submitted confirmation password is treated as a secret and must not be logged.
- Password validation requires a minimum of 8 characters.
- Password validation has no other complexity requirements in this change.
- Password validation must be structured so additional future rules can be added without changing the public reset/login contract.
- Users without a password hash cannot log in through password login.
- Reset-token validity defaults to 3600 seconds.
- Reset-token validity is configurable through `PASSWORD_RESET_TOKEN_EXPIRES_IN_SECONDS`.
- Reset-token validity is configurable through `password_reset_token_expires_in_seconds` in `config/app.yml`.
- `PASSWORD_RESET_TOKEN_EXPIRES_IN_SECONDS` takes precedence over `config/app.yml`.
- Reset-token validity values are expressed in seconds.
- Missing config uses the 3600-second default.
- Non-numeric, zero, or negative reset-token validity config values are rejected during application config loading.
- Reset tokens must be generated from a cryptographically secure random source.
- Persisted reset tokens must not be sufficient to recover raw passwords.
- Reset request captcha uses a first-party challenge generated by the application.
- Captcha must be solved before a reset request creates or replaces a reset token.
- Generated browser bundles under `public/js` remain build artifacts and must not be committed.

## Iteration: Database Migrations And Reset Password Confirmation

This iteration addresses the live reset failure where an existing database has a `users` table without the new `password_hash` column, producing `Unknown column 'password_hash' in 'SET'` during password reset. It adds an explicit database migration mechanism so checked-in schema changes can be applied and tracked for existing databases, then adds a required retype-password confirmation field to the reset-password flow so mistyped new passwords are rejected before password hash updates.

### Added Behavior

- Database setup must support checked-in migration files for schema changes after initial table creation.
- Database setup must always create the migrations tracking table when it does not exist.
- The migrations tracking table creation script must run as an initial bootstrap step during clean startup before application migration discovery.
- Migration files must use a stable naming convention that preserves execution order and makes the affected table clear.
- Migration files must be split by table or by one tightly related table group so table-specific changes are easy to track.
- Existing or newly introduced migration files for this feature must follow the table-scoped naming convention.
- Database setup must create or verify the migrations tracking table before checking application migrations.
- Each migration file must have a stable identifier that is inserted into the migrations table after the migration has executed successfully.
- Migrations already present in the migrations table must be skipped.
- Migrations not present in the migrations table must be executed once and then recorded.
- New schema changes must be represented by a new checked-in migration file instead of relying only on edits to an existing `config/schema/*.sql` create-table file.
- The original schema changes required by this password-login spec must be represented as separate table-scoped migrations rather than one combined migration.
- The users-table migration for this iteration must add `users.password_hash` to existing databases that do not already have it.
- The password-reset-token-table migration for this iteration must create the password reset token table for existing databases that do not already have it.
- Existing `users` tables that lack `password_hash` must receive a nullable `password_hash` column before repositories can update passwords.
- Existing databases that lack the password reset token table must still create it after `users`.
- Migration execution must be idempotent through the migrations table: repeated runs skip already recorded migrations.
- Migration execution must not drop, truncate, rewrite, or reseed existing user, workspace, service, reservation, role, owner, environment, or reset-token data unless a later approved migration explicitly allows it.
- Database setup must be controlled by a feature flag that defaults to enabled.
- When the feature flag is enabled, application startup runs the existing schema setup plus pending migrations before repositories use the database.
- When the feature flag is disabled, application startup must skip migration execution.
- The disabled feature flag mode is intended for future operational flows where migrations are extensive and are run separately.
- The repository must expose an npm migration job that executes pending migrations on demand.
- The npm migration job must run pending migrations regardless of whether application startup migration execution is disabled, so operators can disable startup migrations and run the job explicitly.
- The npm migration job must use the same migrations table and skip already recorded migrations.
- Repository agent guidance must be updated in `AGENTS.md` so future agents preserve the migrations-table, feature-flag, npm-job, and table-scoped migration-file approach.
- Password reset must require a confirmation password value.
- Missing confirmation password must return a deterministic validation error and must not update the password hash or consume the reset token.
- Confirmation password that does not exactly match the submitted password must return a deterministic validation error and must not update the password hash or consume the reset token.
- Successful password reset requires the password and confirmation password to match.
- The reset-password page must show a second password input for confirming the new password.
- The reset-password page must prevent submission and show a deterministic validation error when the two entered password values do not match.

### Preserved Behavior

- Password minimum length remains 8 characters with no additional complexity rules.
- Raw password and confirmation password values remain unlogged and unpersisted.
- Existing reset-token validation, single-active-token, consumption, and no-JWT-on-reset behavior remain unchanged after confirmation succeeds.
- Reset URL delivery remains log-only with the existing TODO and must not be returned in API responses.

## Deterministic Behavior

### Password Login

- `POST /api/login` trims and lowercases the submitted email.
- Missing or blank email returns `400` with `Email required`.
- Missing or blank password returns `400` with `Password required`.
- Unknown email returns `403` with the same failure shape used for invalid credentials.
- Existing email with missing password hash returns `403` with the same failure shape used for invalid credentials.
- Existing email with wrong password returns `403` with the same failure shape used for invalid credentials.
- Existing email with correct password returns `200` with the existing JWT login success contract:
  - `ok: true`
  - the current `user` object
  - `token`
  - `token_type: "Bearer"`
  - `expires_in_seconds`
- A successful password login does not expose password hash state.
- The frontend login page requires both email and password.
- The frontend login page stores the returned JWT exactly as the current bearer login flow does.

### Captcha Challenge

- The application exposes an unauthenticated API that creates a first-party captcha challenge for the reset-password request flow.
- The challenge response includes a challenge identifier and displayable prompt data.
- The challenge response does not include the answer.
- Captcha answers are validated server-side.
- Missing, unknown, expired, or incorrect captcha validation fails the reset request.
- Captcha validation is single-use for successful reset requests.
- Captcha failures do not create reset tokens.
- Captcha challenge lifetime and storage must be deterministic and testable; if the implementation uses in-memory storage, that storage is explicitly process-local and resets on server restart.

### Reset Link Request

- The frontend exposes a forgot-password/reset-link request flow reachable from the login page.
- The reset-link request accepts email plus captcha challenge data.
- The reset-link request trims and lowercases email.
- Missing or blank email returns a deterministic validation error.
- Missing or invalid captcha returns a deterministic validation error.
- After valid captcha, the reset-link request returns a generic success response regardless of whether the email belongs to an existing user.
- Unknown emails do not create reset tokens.
- Existing users receive exactly one active reset token:
  - any previous active token for that user is invalidated before or when the new token is created
  - expired or used historical tokens remain persisted for auditability unless cleanup is added by a later approved spec
- The server logs the reset URL for existing users.
- The log line must include a TODO that the logged URL must be removed when email delivery is implemented.
- The reset URL is not returned in the API response.
- The frontend shows the same generic success state after a valid reset-link request whether the email exists or not.

### Reset Token Validation Page

- The application serves a reset-password frontend route at `/reset-password/:token`.
- The reset-password page validates the token before showing the new-password form.
- Missing, unknown, expired, or used tokens show a deterministic invalid-token state and do not show an enabled password-reset form.
- A valid token shows a password-reset form.
- Token validation does not mark the token as used.

### Password Reset

- The password-reset submit API accepts a reset token, a new password, and a confirmation password.
- Missing token returns a deterministic validation error.
- Missing password returns a deterministic validation error.
- Missing confirmation password returns a deterministic validation error.
- Confirmation password that does not exactly match the new password returns a deterministic validation error.
- Passwords shorter than 8 characters return a deterministic validation error.
- Unknown, expired, or used tokens return a deterministic invalid-token error.
- A successful password reset:
  - stores a new password hash for the token's user
  - marks the used reset token as used for that user
  - invalidates any other active reset token for that user
  - returns a generic success response
- Used tokens cannot be used again.
- After successful reset, the frontend sends the user back to the login flow or provides a clear path to login.
- Successful reset does not automatically issue a JWT.

## Assumptions

- The request is backend-led but includes required frontend support for the login, forgot-password, and reset-password flows.
- Password login replaces email-only login for browser and API clients.
- First-party captcha is acceptable for this phase because external captcha providers and keys are out of scope.
- Reset URLs are logged only as a temporary development/operational bridge until email delivery is implemented.
- Preventing email enumeration is required for reset requests, so reset-link request success is generic after captcha passes.
- Single active reset token per user means creating a new token invalidates any previous active token for that same user.
- Existing users will need a reset-password token to set their first password unless a later approved seed or migration workflow sets password hashes.

## Impact And Regression Considerations

- Existing API clients that call `POST /api/login` with only email will fail until updated to send a password.
- Existing users without password hashes will be unable to log in until they complete reset-password.
- User persistence changes affect schema creation, database migration execution for existing databases, repository mapping, entity shape, and integration tests.
- Reset-token persistence adds a new schema object and startup schema ordering requirement.
- Migration execution must be conservative because the app manages runtime workspace and user data in MySQL; this iteration's migration must not erase existing data.
- Captcha storage and expiration must not create nondeterministic test failures.
- Generic reset success reduces email enumeration risk but can make local troubleshooting depend on server logs for existing-user reset URLs.
- Password hashing parameters must balance security with deterministic test runtime.
- The login page must remain usable at mobile and desktop widths after adding password, forgot-password, captcha, and reset-password UI states.
- The reset-password page must remain usable at mobile and desktop widths after adding the confirmation password field.
- JWT renewal, bearer API calls, logout, workspace authorization, reservation behavior, and admin authorization should remain unchanged after successful authentication.

## Acceptance Criteria

- `POST /api/login` requires email and password.
- `POST /api/login` rejects missing email with `400`.
- `POST /api/login` rejects missing password with `400`.
- `POST /api/login` rejects unknown email, missing password hash, and wrong password with `403`.
- `POST /api/login` returns the existing JWT success payload for a correct password.
- Raw passwords are not persisted or logged.
- Persisted users can store and verify password hashes.
- A migrations tracking table exists after database setup.
- The migrations tracking table is created automatically on clean startup before application migration discovery.
- Pending checked-in migrations are detected by comparing migration identifiers to the migrations tracking table.
- Migration file names preserve deterministic order and identify the affected table.
- Migration files are split per table unless a migration must update one tightly related table group atomically.
- The original password-login schema changes are split into separate table-scoped migrations.
- Pending migrations run once and are recorded after success.
- Already recorded migrations are skipped on later startup or migration-job runs.
- Existing databases with a `users` table but no `password_hash` column are repaired on startup without data loss.
- Existing databases without the password reset token table create that table through its own migration.
- The password-hash migration is idempotent when the column already exists.
- Startup migration execution defaults to enabled.
- When the startup migration feature flag is disabled, startup skips pending migrations.
- An npm migration job can execute pending migrations explicitly.
- Reset-token validity defaults to 3600 seconds.
- Reset-token validity can be configured by `password_reset_token_expires_in_seconds`.
- Reset-token validity can be configured by `PASSWORD_RESET_TOKEN_EXPIRES_IN_SECONDS`.
- The environment variable takes precedence over file config.
- Invalid reset-token validity config is rejected.
- Captcha challenge creation returns a challenge id and prompt without the answer.
- Reset-link request requires a valid captcha before token creation.
- Reset-link request returns generic success after valid captcha for both known and unknown emails.
- Known-user reset-link request persists one active reset token and invalidates any previous active token for that user.
- Unknown-user reset-link request persists no token.
- Known-user reset-link request logs `/reset-password/:token` with a TODO to remove logging when emails are implemented.
- Reset URL is not returned in reset-link request API responses.
- `/reset-password/:token` frontend route validates the token before enabling password reset.
- Valid reset tokens allow setting a new password that is at least 8 characters long.
- Password reset requires retyping the new password in a confirmation field.
- Password reset rejects missing confirmation password with deterministic validation.
- Password reset rejects non-matching password and confirmation password with deterministic validation.
- Rejected confirmation validation does not update password hashes, mark tokens used, or invalidate active tokens.
- Short passwords are rejected with deterministic validation.
- Successful password reset marks the token used and invalidates any other active token for that user.
- Used, expired, unknown, or missing tokens cannot reset passwords.
- Successful password reset does not issue a JWT.
- Frontend login supports email plus password.
- Frontend forgot-password flow supports captcha and generic success.
- Frontend reset-password page supports token validation, invalid-token state, new-password submission, and confirmation-password match validation.
- README documents password login, reset-token config, temporary logged reset URLs, and future email replacement.
