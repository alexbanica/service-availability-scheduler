# SPEC: Workspace Environments And UUID Service Contracts

Status: Approved

## Purpose

Correct service creation failures caused by missing `service_key` values and align workspace, service, environment, owner, and user contracts around backend-generated UUIDs and workspace-scoped reference definitions.

## Problem Statement

Creating a service can fail with `field service_key doesn't have a default value` because service-environment association rows require a unique `service_key`, but the creation path is not consistently guaranteed by the persisted schema and backend behavior.

Service creation also currently mixes responsibilities that should belong to Workspace Management:

- defining which environments exist in a workspace
- defining which owners exist in a workspace
- selecting which existing workspace environments apply to a service
- selecting which existing workspace owner applies to a service

The requested behavior separates those responsibilities. Workspace Management owns environment and owner definitions per workspace. Service Management only creates or edits services by selecting from environments and owners already available in the selected workspace. Numeric ids must be removed from the database and application model for workspaces, services, environments, owners, users, and their relationships. UUIDs are the canonical identifiers.

## Relationship To Existing Specs

This spec amends and supersedes conflicting clauses in:

- `specs/SPEC-admin-workspace-management-ui.md`
- `specs/SPEC-admin-service-management-ui.md`

Specifically, this spec replaces any prior behavior that allowed Service Management create or edit forms to create missing environment or owner values. It also replaces prior schema, API, and UI identity assumptions that use numeric workspace, service, environment, owner, or user ids.

## Scope

- Replace numeric database identifiers for workspaces, services, environments, owners, users, reservations, roles, memberships, and invitations with UUID identifiers.
- Add or complete backend-generated UUID identity for workspaces and users.
- Preserve existing service and environment UUID identity concepts while making those UUIDs the primary persisted identifiers.
- Ensure workspace, service, environment, owner, and user identifiers are UUID strings throughout schema, backend contracts, frontend state, and UI actions.
- Ensure `service_key` is always generated by backend-controlled logic when a service is associated with an environment.
- Keep `service_key` globally unique.
- Add Workspace Management UI/API behavior for creating, listing, and deleting environments per workspace.
- Add Workspace Management UI/API behavior for creating, listing, and deleting owners per workspace.
- Add Workspace Management action buttons for `Invite user`, `Create owner`, and `Create environment` on each administered workspace.
- Update Workspace Management overview stats to include users, services, owners, and environments per administered workspace.
- Add stat-tag popups that show the clicked workspace's users, services, owners, or environments.
- Update Service Management create and edit flows so admins can select only existing environments from the selected workspace.
- Update Service Management create and edit flows so admins can select only existing owners from the selected workspace.
- Prevent Service Management from creating new environments.
- Prevent Service Management from creating new owners.
- Ensure services and environments are scoped to their owning workspace and are not shareable across workspaces.
- Ensure owners are scoped to their owning workspace and are not shareable across workspaces.
- Preserve current reservation behavior except where UUID identifiers are required.
- Recreate the database schema instead of preserving or migrating existing database rows.
- Rewrite database schema/migration files for an empty database using the new UUID schema as the source of truth.

## Out Of Scope

- Cross-workspace sharing of services or environments.
- Cross-workspace sharing of owners.
- Global environment catalogs.
- Global owner catalogs.
- Changing reservation claim, extend, or release workflows beyond required identifier compatibility.
- Migrating historical data.
- Preserving existing database rows, historical reservations, legacy numeric ids, or compatibility with old schema versions.
- Role model changes.
- Workspace transfer or ownership changes.
- Bulk environment or service import/export.
- Bulk owner, user, service, or environment editing from stat popups.
- Email or notification behavior.

## Definitions

- `UUID`: A backend-generated opaque identifier exposed by API responses, frontend state, URLs, and UI actions for workspaces, services, and users.
- `Workspace UUID`: The external identifier for a workspace.
- `User UUID`: The external identifier for a user.
- `Service UUID`: The external identifier for a service.
- `Environment UUID`: The external identifier for an environment.
- `Owner UUID`: The external identifier for an owner.
- `Service key`: The unique backend-generated reservation key for one service-environment association.
- `Workspace environment`: An environment row owned by exactly one workspace.
- `Workspace owner`: An owner row owned by exactly one workspace.
- `Service environment association`: A row connecting one service to one environment in the same workspace.
- `Workspace overview stat tag`: A clickable count tag in Workspace Management for users, services, owners, or environments.

## Inputs And Constraints

- The current schema has internal numeric primary keys for `workspaces`, `users`, `services`, `environments`, and association tables, but those numeric identifiers must be replaced.
- The current service owner is a nullable text field on `services`; the target behavior requires workspace-scoped owner records.
- The current schema already has UUID-like string identifiers for services and environments.
- The current schema does not yet define UUID primary identifiers for workspaces and users.
- The target schema may be recreated from scratch; no data migration or backward-compatible schema evolution is required.
- The implementation target is an empty database provided by the user.
- Existing schema/migration files must be rewritten to create the UUID-based target schema directly.
- `service_environments.service_key` remains `NOT NULL` and globally unique in the target schema.
- Reservations reference `service_key`.
- Existing Service Management work in the current branch uses selected-workspace service management and service edit flows.
- Backend code must preserve onion architecture dependency direction:
  - controllers translate HTTP input/output
  - services enforce authorization and use-case behavior
  - repositories own MySQL persistence details
  - domain entities and DTOs stay independent from Express, MySQL, browser DOM, and filesystem/runtime details
- UI changes must remain consistent with the existing admin-dashboard style and responsive behavior.

## Deterministic Behavior

### Workspace Overview Stats

- Workspace Management shows each administered workspace with count tags for:
  - users
  - services
  - owners
  - environments
- User count is the number of workspace membership rows for the workspace.
- Service count is the number of service rows for the workspace.
- Owner count is the number of owner rows for the workspace.
- Environment count is the number of environment rows for the workspace.
- Counts are loaded from backend data scoped to the signed-in admin's administered workspaces, not inferred from partially loaded client-side state.
- Count tags are visually clickable and keyboard reachable.
- Clicking a count tag opens a popup for the clicked workspace and clicked resource type.
- The popup title identifies both the workspace and resource type, for example `Owners in Payments`.
- The popup contents match the clicked tag:
  - users tag shows workspace users
  - services tag shows workspace services
  - owners tag shows workspace owners
  - environments tag shows workspace environments
- The popup does not mix data from other resource types.
- The popup does not show data from another workspace.
- If the clicked resource type has no rows, the popup shows an empty state for that resource type.
- Closing the popup clears the selected workspace, resource type, loaded data, loading state, and error state for the popup.
- Popup data loading errors remain scoped to the popup.
- The popup is read-only in this spec; create/delete owner and environment actions remain in their dedicated Workspace Management controls.

### Workspace Management Actions

- Each administered workspace row or card shows action buttons for:
  - `Invite user`
  - `Create owner`
  - `Create environment`
- The action buttons are displayed alongside each other in the workspace management action area.
- Pressing `Invite user` opens the existing invite-user popup for that workspace.
- Pressing `Create owner` opens a create-owner popup for that workspace.
- Pressing `Create environment` opens a create-environment popup for that workspace.
- Each popup title identifies the target workspace and action.
- Each popup has its own form state, validation error state, submitting state, and close/cancel behavior.
- Closing or canceling any popup resets only that popup's state.
- Successful owner creation closes the create-owner popup, refreshes owner counts and owner lists for that workspace, and makes the owner available in Service Management for that workspace.
- Successful environment creation closes the create-environment popup, refreshes environment counts and environment lists for that workspace, and makes the environment available in Service Management for that workspace.
- The create-owner popup does not create environments or users.
- The create-environment popup does not create owners or users.
- The invite-user popup does not create owners or environments.

### UUID Identity

- The database schema uses UUID string columns as primary identifiers for workspaces, users, services, environments, reservations, workspace memberships, workspace invitations, and user roles.
- The database schema uses UUID string columns as primary identifiers for owners.
- Join tables and foreign keys reference UUID identifiers, not numeric internal ids.
- API responses for workspaces expose `workspace_id` as the workspace UUID.
- API responses for users expose `user_id` as the user UUID.
- API responses for services expose `service_id` as the service UUID.
- API responses for environments expose `environment_id` as the environment UUID.
- API responses for owners expose `owner_id` as the owner UUID.
- Frontend entities and controller state use UUID strings for workspace, service, environment, owner, and user identity.
- Frontend API calls address workspaces by workspace UUID, services by service UUID, and users by user UUID wherever those resources are path parameters or request identity fields.
- Numeric database ids are not present in target schema, API responses, request bodies, frontend state, or UI rendering.
- Session authentication stores the authenticated user's UUID or a session-local value that resolves to a UUID-backed user without relying on numeric user ids.

### UUID Creation And Uniqueness

- New users receive a backend-generated `user_id` UUID before persistence.
- New workspaces receive a backend-generated `workspace_id` UUID before persistence.
- New services receive a backend-generated `service_id` UUID before persistence.
- New environments receive a backend-generated `environment_id` UUID before persistence.
- New owners receive a backend-generated `owner_id` UUID before persistence.
- Workspace UUIDs are globally unique.
- User UUIDs are globally unique.
- Service UUIDs are globally unique.
- Environment UUIDs are globally unique.
- Owner UUIDs are globally unique.
- A service belongs to exactly one workspace and cannot be selected, edited, deleted, reserved, or listed through another workspace.
- An environment belongs to exactly one workspace and cannot be selected for a service in another workspace.
- An owner belongs to exactly one workspace and cannot be selected for a service in another workspace.

### Service Key Generation

- The backend always supplies or generates `service_key` for service-environment association inserts.
- Service creation and service edit never rely on a missing database default for `service_key`.
- Database triggers or compatibility migration helpers are not required for old-schema compatibility.
- Application behavior must explicitly generate and persist `service_key`.
- `service_key` is unique across all service-environment associations.
- `service_key` is stable for an existing service-environment association.
- Re-associating an existing service with an existing environment reuses the existing association and does not create a duplicate key.
- Creating a new service-environment association generates exactly one unique `service_key`.
- Reservations continue to use `service_key` to identify a claimable service environment.

### Workspace Environment Management

- Workspace Management shows each administered workspace's environments or provides an environment management action for each administered workspace.
- Workspace admins can create environments for a workspace from Workspace Management.
- Workspace admins can delete environments from Workspace Management only when deletion will not break referential integrity or active service associations.
- Environment names are trimmed before persistence.
- Creating an environment with a blank name prevents the API call or returns a validation error equivalent to `Environment name is required.`
- Creating a duplicate environment name within the same workspace is rejected case-insensitively with a clear duplicate-environment error.
- The same environment name may exist in different workspaces because environments are workspace-scoped.
- Non-admin workspace members cannot create or delete workspace environments.
- Environment list ordering is deterministic, alphabetical by environment name unless an existing repository order already guarantees this.

### Workspace Owner Management

- Workspace Management shows each administered workspace's owners or provides an owner management action for each administered workspace.
- Workspace admins can create owners for a workspace from Workspace Management.
- Workspace admins can delete owners from Workspace Management only when deletion will not break referential integrity or active service references.
- Owner names are trimmed before persistence.
- Creating an owner with a blank name prevents the API call or returns a validation error equivalent to `Owner name is required.`
- Creating a duplicate owner name within the same workspace is rejected case-insensitively with a clear duplicate-owner error.
- The same owner name may exist in different workspaces because owners are workspace-scoped.
- Non-admin workspace members cannot create or delete workspace owners.
- Owner list ordering is deterministic, alphabetical by owner name unless an existing repository order already guarantees this.

### Service Creation

- Service Management requires a selected workspace before service creation controls are shown.
- Creating a service asks for a service name.
- The service name maps to the persisted service label.
- The backend generates the service UUID; the browser does not submit a service UUID for new services.
- The service create form lets the admin select one or more environments only from the selected workspace's existing environment list.
- The service create form lets the admin select zero or one owner only from the selected workspace's existing owner list.
- The service create form does not provide free-text creation of new environment names.
- The service create form does not provide free-text creation of new owner names.
- If the selected workspace has no environments, Service Management shows an empty-state message directing the admin to define environments in Workspace Management, and service creation is disabled.
- Submitting with no selected environments prevents the API call and shows `Select at least one environment.`
- The backend rejects any submitted environment UUID that does not belong to the selected workspace.
- The backend rejects any submitted owner UUID that does not belong to the selected workspace.
- Successful service creation creates service-environment associations only for the selected workspace environments.
- Successful service creation returns the generated service UUID.
- Successful service creation refreshes the selected workspace service list.

### Service Editing

- Service editing identifies the target service by service UUID.
- The edit form lets the admin select one or more environments only from the selected workspace's existing environment list.
- The edit form lets the admin select zero or one owner only from the selected workspace's existing owner list.
- The edit form does not create new environments.
- The edit form does not create new owners.
- Editing environments replaces the service's association set with the selected existing environments.
- Editing owner replaces the service's owner reference with the selected existing owner, or clears the owner when no owner is selected.
- The backend rejects any submitted environment UUID that does not belong to the selected workspace.
- The backend rejects any submitted owner UUID that does not belong to the selected workspace.
- Removed associations are deleted only for the edited service.
- The edited service keeps the same service UUID.
- Existing reservations tied to removed service-environment associations are out of scope unless database constraints require deterministic cleanup.

### Workspace Isolation

- Workspace, service, and environment lookup by UUID always verifies workspace membership or workspace admin authorization as applicable.
- Owner lookup by UUID always verifies workspace membership or workspace admin authorization as applicable.
- A service UUID that belongs to another workspace returns `Service not found` or the existing not-authorized response, never a cross-workspace mutation.
- An environment UUID that belongs to another workspace returns a validation or not-found error and is never associated with the selected workspace's service.
- An owner UUID that belongs to another workspace returns a validation or not-found error and is never assigned to the selected workspace's service.
- Workspace member service listing includes only services for workspaces where the signed-in user is a member.
- Workspace admin environment management includes only workspaces where the signed-in user is an admin.

### Database Recreation

- The implementation assumes the user provides an empty database.
- Database schema/migration files are rewritten to create the UUID-based target schema directly.
- Target schema files define UUID identifiers directly and must not include incremental ALTER compatibility for existing numeric-id tables.
- Startup schema initialization applies cleanly to an empty database.
- No existing rows, reservations, users, workspaces, services, environments, owners, invitations, memberships, or roles need to be preserved.
- Seed data, if present, must use UUID identifiers or deterministic backend-generated UUID-compatible values.
- Existing frontend references to numeric `id`, `workspaceId`, `adminUserId`, or `claimedById` are replaced with UUID-based fields.

## Assumptions

- Numeric primary keys must be removed from the intended MySQL schema and application model.
- The phrase "We don't use id's anymore" means UUID identifiers replace numeric ids across schema, backend contracts, frontend state, and UI behavior.
- `service_key` may remain a separate reservation key even though services also have UUIDs.
- Environment deletion from Workspace Management should be blocked while the environment is associated with any service, unless implementation discovers an existing cascade contract that already handles this safely.
- Owner deletion from Workspace Management should be blocked while the owner is assigned to any service, unless implementation discovers an existing clear-or-cascade contract that already handles this safely.
- Existing session authentication must be adjusted away from numeric user ids.

## Impact And Regression Considerations

- This change affects API contracts, frontend state keys, tests, schema design, and authorization lookup paths.
- Existing admin workspace-management and service-management implementation work in the current branch must be reconciled with this spec before implementation continues.
- Service Management environment tag/free-text behavior from the prior approved spec becomes invalid for create/edit environment selection.
- Service Management free-text owner behavior becomes invalid for create/edit owner selection.
- Reservation grouping and claim/release behavior must continue to work because reservations still depend on `service_key`.
- Existing local databases can be recreated, so implementation should prioritize a clean UUID schema over compatibility with existing numeric-id data.
- Migration/schema work must target the empty-database path and avoid compatibility code for old numeric-id schemas.
- UUID lookup paths must not leak cross-workspace resource existence.
- Browser state maps currently keyed by numeric workspace ids must be changed carefully to avoid stale state, selection loss, and cache collisions.

## Acceptance Criteria

- Creating a service never fails because `service_key` lacks a database default.
- The target schema contains no numeric auto-increment primary keys for application entities or relationship tables.
- Schema/migration files create the UUID-based schema directly on an empty database.
- Service Management cannot create an environment.
- Service Management cannot create an owner.
- Service Management can only select environments already defined for the selected workspace.
- Service Management can only select owners already defined for the selected workspace.
- Workspace Management can define environments per workspace.
- Workspace Management can define owners per workspace.
- Workspace Management shows `Invite user`, `Create owner`, and `Create environment` actions alongside each administered workspace.
- Each workspace action opens the correct popup and creates only the intended resource.
- Workspace Management overview shows user, service, owner, and environment count tags for each administered workspace.
- Clicking a count tag opens a popup listing only that clicked workspace's resources for the clicked tag type.
- A service created in one workspace cannot be used from another workspace.
- Workspace, service, and user identifiers visible in API responses and browser state are UUID strings.
- Numeric ids are not present in schema, UI, browser state, or browser API calls.
- Existing service reservations continue to use valid service keys.
- Existing automated tests are updated to cover UUID identity, workspace isolation, service key generation, environment selection restrictions, and owner selection restrictions.
